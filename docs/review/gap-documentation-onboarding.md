# Documentation / Onboarding Gap Analysis

**Dimension:** `documentation-onboarding`
**Scope:** The quality, accuracy, discoverability, and completeness of ProofForge’s user and contributor documentation, project scaffolding, first-run experience, and translated materials.
**Date:** 2026-07-10

---

## Executive summary

ProofForge has a rich, well-organized doc tree (INDEX, onboarding, tutorials, RFCs, per-target notes, validation gates, and a machine-readable backend-status contract). The mechanical doc↔code sync audit is clean, the project scaffold works, and the primary product gate passes. However, the very first command shown to new users is wrong, the Chinese translation layer has hundreds of broken internal links and visible formatting glitches, the translation sync is 4 docs behind, and there is no release-level narrative (CHANGELOG/release notes). For a production-grade compiler/CLI, the onboarding dimension still has sharp edges that will frustrate external adopters.

**Maturity score:** **6 / 10**

- Strong points: doc structure, sync audit, working scaffold, product tutorials, CI docs gate.
- Weak points: incorrect CLI examples, broken i18n links, translation lag, missing release notes, CLI help discoverability.

## Gaps

### 1. First-run CLI examples use the wrong `--module` value

| | |
|---|---|
| **Area** | README / AGENTS onboarding commands |
| **Evidence** | `README.md:103` and `AGENTS.md:73` show:<br>`lake env proof-forge build --target evm --root . --module contract -o build/evm/Counter.bin Examples/Product/Counter.lean`<br>`README.md:103` also shows the same `--module contract` with `Examples/Backend/Evm/Contracts/Counter.lean`.<br>Running either fails with:<br>`uncaught exception: no spec : ProofForge.Contract.ContractSpec found while loading module contract; define one with contract_source or def spec : ProofForge.Contract.ContractSpec`.<br>The actual module namespace in both files is `Counter` (see `Examples/Backend/Evm/Contracts/Counter.lean:9` and `Examples/Product/Counter.lean:10`).<br>The same incorrect `--module contract` pattern is copied into `docs/validation-gates.md:60`, `docs/zh/validation-gates.zh.md:57`, `docs/development-log.md:10554`, `docs/superpowers/specs/2026-07-06-psy-metadata-hardening-design.md:101`, `docs/zh/README-root.zh.md:96`, and `docs/zh/targets/evm.zh.md:103`. |
| **Severity** | **Blocker** |
| **Remediation** | Replace `--module contract` with `--module Counter` (or omit `--module` where auto-resolution works) in every English doc example, then resync translations. Add a CI docs check that runs the literal README/AGENTS build commands and fails on copy-paste drift. |

### 2. Chinese documentation contains hundreds of broken internal links and formatting defects

| | |
|---|---|
| **Area** | i18n docs (`docs/zh/*.md`) |
| **Evidence** | A link scan of `docs/zh/` found **314 likely broken internal links**. Representative samples:<br>`docs/zh/onboarding.zh.md:83` links to `validation-gates.md` (resolves in `docs/zh/`, where only `validation-gates.zh.md` exists).<br>`docs/zh/decisions.zh.md` links to `zh/review-checklist.md`, `rfcs/0005-solana-sbpf-assembly-backend.md`, `targets/solana-sbpf-asm.md`, etc., from inside `docs/zh/` — most targets do not exist at those relative paths or are missing the `.zh.md` suffix.<br>`docs/zh/targets-README.zh.md` links to bare `move-family.md`, `evm.md`, etc., which do not exist in `docs/zh/targets/`.<br>`docs/zh/INDEX.zh.md:76` has a Markdown formatting glitch where `工程-` is merged with the next bullet (`- [开发标准]`).<br>`docs/zh/INDEX.zh.md:108` has another merged-line defect (`[Cloudflare Workers 目标]` stuck to the previous bullet). |
| **Severity** | **High** |
| **Remediation** | Run a link checker over `docs/zh/` in CI (e.g., extend `scripts/i18n/check-sync.sh` or add `lychee`/`markdown-link-check`). Fix relative-path convention so links from `docs/zh/` point to sibling `.zh.md` files or to `../<name>.md` for English fallbacks consistently. Re-run the translation script after fixing the English source formatting so the glitch is not propagated. |

### 3. Translation sync is 4 docs behind

| | |

### 4. No CHANGELOG or release notes

### 5. CLI help is hard to discover and subcommands do not accept `--help`

| | |
|---|---|
| **Area** | CLI discoverability |
| **Evidence** | `lake env proof-forge check --help` and `lake env proof-forge emit --help` both respond with `unknown option: --help` and dump the full legacy usage list (187 lines in `ProofForge/Cli/Usage.lean`). Only `proof-forge deploy --help` prints a focused subcommand help.<br>The top-level usage mixes product commands with dozens of legacy `--emit-*-ir-*` / `--learn-*` flags, burying the preferred `build|emit|check --target` surface. |
| **Severity** | **Medium** |
| **Remediation** | Make `proof-forge <subcommand> --help` work for `build`, `check`, `emit`, and `init`. Split legacy fixture flags into a separate `proof-forge --help-internal` or `proof-forge emit --list-fixtures` page so the default help shows the product path first. |

## Top 5 gaps (prioritized)

1. **CLI examples use the wrong `--module contract`** — the first command in README/AGENTS fails for external users. This is the single biggest onboarding blocker.
2. **Hundreds of broken links in Chinese docs** — `docs/zh/` links point to non-existent `.md` files or use wrong relative paths, undermining the translated onboarding path.
3. **Translation sync lag** — 4 authoritative English docs are out of sync with their Chinese translations, causing `just docs-check` to fail.
4. **No CHANGELOG or release tags** — users have no migration/upgrade narrative even though the versioning RFC exists.
5. **Subcommand `--help` missing and default help is overwhelmed by legacy flags** — everyday users cannot easily discover the preferred `build|emit|check --target` surface.

---

## Overall maturity score: 6 / 10

ProofForge’s documentation is extensive, well-structured, and mostly accurate for contributors already inside the monorepo. The mechanical sync guard, working scaffold, and passing product gate are real strengths. To reach production-grade onboarding, the highest-leverage fixes are: (1) correct the README/AGENTS CLI examples, (2) repair the Chinese link layer and resync translations, and (3) add a focused subcommand help surface. Those three changes would raise this dimension to 7–8.
